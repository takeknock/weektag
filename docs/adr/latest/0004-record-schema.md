# 0004. レコードスキーマ：構造化タグ・ローカル時刻・ミニULID

## ステータス
Accepted

## コンテキスト
タグの持ち方が入力UX・集計・エクスポートの全消費者を縛る。TimeTagger 式のインライン #タグ（説明文中から都度抽出）は、レポート・CSV・エージェントの全員に毎回のテキスト再パースを強いる。また bash は行中の `#` 以降をコメント扱いするため、CLI 入力に裸の `#` を使う設計は事故る。

## 決定
1行のスキーマ：

```json
{"id":"01JZK3AB","start":"2026-07-06T09:00:00+09:00","stop":"2026-07-06T10:30:00+09:00","tags":["writing","client-a"],"note":"ブログ下書き"}
```

- `tags` は文字列配列、`note` は自由記述の別フィールド。実行中レコードは `stop` キー自体を持たない
- 時刻は**ローカル時刻＋UTCオフセット**（ISO 8601）で格納・表示。週帰属の判定はローカル日付で行う。UTC 正規化はしない（JST 深夜の記録が前日15時Zになる不可読性を排除。オフセット付きなので変換は常に可能）
- `id` は時系列ソート可能な**ミニULID**（タイムスタンプ＋乱数、自作）。コマンドでの指定は前方一致を許す
- CLI 入力構文：タグは位置引数（`tt start writing client-a`）、note は `-m`。`'#writing'` とクォート付きで来た場合は `#` を剥いで受理する

## 結果
- jq やエージェントが `group_by(.tags)` 等で即集計できる。タグのリネームも機械的に安全
- 「顧客 × 作業種別」のようなクロス集計が配列タグで自然に表現できる
- CSV/TSV の列（tags / note）へ素直にマップでき、Excel 転記が単純になる
